MacSessionGuard can never see “unlocked” — three fixes, one chosen

Repo: open-codex-computer-use Branch: fix/mac-session-guard-lock-detection Plan: plans/260720-1024-mac-session-guard-lock-detection/plan.md

On macOS 26 the CGSSessionScreenIsLocked key is simply omitted from CGSessionCopyCurrentDictionary() when the machine is genuinely unlocked; it appears (as true) only while locked. The guard treats an absent key as “locked”, so under the default fail-closed policy every Computer Use tool call is blocked even on an unlocked machine — verified empirically with a standalone probe.

Chosen
Proposal B — console-gated absent-key fix. Absent lock key reads as unlocked only when kCGSSessionOnConsoleKey corroborates that this session owns the console. Auto-picked per the judge's recommendation under --auto; decision logged in the auto-decisions report.

The three directions

Runner-up

A · Minimal Apple-semantics flip

Non-empty dict + absent key → unlocked. One branch flips; nothing else changes.

  • Smallest possible diff (~15–20 LOC source)
  • Matches observed macOS behavior exactly
  • Weakness: bare inference — reports UNLOCKED for a fast-user-switched-away session (key absent, off-console), a real false-unlocked under the default policy
~65–95 LOC total · single signal
Chosen

B · Console-gated fix

Absent key → unlocked only if kCGSSessionOnConsoleKey coerces true. Off-console, missing, or garbage stays fail-closed. Explicit lock key is always checked first and always wins.

  • Strictly more conservative than A — only ever adds fail-closed cases
  • Closes the fast-user-switch false-unlocked gap for ~10 extra LOC
  • Degrades gracefully on OS versions where absent means something else
~25–35 LOC source · corroborated signal
Rejected

C · Multi-signal notification tracking

Cross-check the dict against com.apple.screenIsLocked distributed notifications held in a stateful listener.

  • 150–220 LOC, new abstraction + fake, run-loop dependency
  • Silently degrades to A/B in contexts without a run loop
  • Fatal: introduces a new false-unlocked path via stale or missed notifications — its own author recommended against it
~150–220 LOC · new attack surface

Judge (Opus): B is strictly more conservative than A and never more permissive; it answers A's own open question (“is a corroborating second signal worth it?”) with a low-cost yes. C is a KISS/YAGNI reject. B's console gate does not defend against a hypothetical future OS that omits the key while locked — that residual is bounded by log-based detection and the unchanged opt-in escape hatch, and is accepted in writing in the plan.

Cache policy changes with it: shouldCache never caches an unlocked snapshot, so a lock transition can never be masked by the 200 ms TTL — stale locked for ≤200 ms after unlocking costs at most one spurious block, never a false unlock.

Gate history

Round 1 — BLOCKED. Red-team: stale-unlocked cache across a lock transition (TTL window); FUS/on-console correlate never verified in the safe direction; fail-open reachable on untested macOS 14–15. Validator: README line 42 still documents absent = locked as intended behavior.
Revisions (R1–R5). Never-cache-unlocked policy with a testable shouldCache seam; mandatory FUS-via-SSH human verification in the PR checklist; residual accepted in writing with one-shot log-based detection (an OS-version gate was rejected — it would keep the guard permanently broken on macOS 14–15); README correction promoted to required scope; explicit-lock-wins pinned off-console too.
